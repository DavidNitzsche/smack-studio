Critical — Phase 1 Blockers
01
Multi-tenancy
Critical
Hard blocker. The app currently operates as a single-tenant system where all of Workprint's data lives in one SQLite database with no studio-level isolation. If a second studio signed up today, they would have access to every project, invoice, contractor, and booking Workprint has ever created. This cannot ship.
Item Current State What's Needed Phase Complexity
Database architectureIsolation model Single workprint.db SQLite file. All data belongs to one implicit "studio." No studio_id column on any table. Zero row-level scoping. Per-studio SQLite files (/data/{studio_id}.db) is the recommended path — maintains the existing SQLite stack, provides hard data boundaries, and allows studio-level backup/restore. Alternatively: PostgreSQL with row-level security (RLS) — more operationally complex but supports shared connection pooling. Phase 1 XL
All API routesRequest scoping Every API route (/api/projects, /api/invoices, /api/crew, etc.) queries the single database with no studio filter. Any authenticated user sees all data. All API routes must derive studio_id from the authenticated session and scope every query to that studio's database or rows. This affects every single route in the app — treat it as a full API audit. Phase 1 XL
IDs & referencesCross-studio leakage Project IDs, contractor IDs, and invoice IDs are sequential integers (1, 2, 3...). These can be guessed and accessed across studios if routes aren't properly scoped. Migrate to UUIDs (or nanoids) for all public-facing IDs. Even with proper scoping, sequential integers are an unnecessary information disclosure risk. Enforce ownership checks on every resource fetch — WHERE id = ? AND studio_id = ?. Phase 1 L
Studio recordEntity model No studios table exists. The concept of "a studio" doesn't exist in the data model — it's all just Workprint. Create a studios table with: id, name, slug, logo_url, timezone, plan, created_at. Every other entity (users, projects, invoices) must have a studio_id foreign key. Phase 1 L
02
Authentication
Critical
Hard blocker. Auth is currently hard-coded to allow only @workprint.la Google Workspace accounts. There is no self-serve signup, no password reset, no email verification, and no way for any external user to create an account.
Item Current State What's Needed Phase Complexity
Auth providerCore identity layer Custom Next.js auth middleware that validates Google OAuth and checks the email domain against a hard-coded @workprint.la allowlist. Anyone without that domain is rejected. Replace entirely with Clerk.com. Clerk provides: email/password auth, Google OAuth, magic links, email verification, password reset, session management, and an organizationId concept that maps cleanly to studios. Use Clerk's custom UI appearance API to match Smack's design — never use Clerk's hosted pages. Phase 1 XL
MiddlewareRoute protection All protected routes check for the Workprint Google session. Middleware is tightly coupled to the Google Workspace integration. Replace all auth middleware with Clerk's clerkMiddleware(). Route protection becomes declarative. Session contains userId and orgId (studio). Every API handler reads from auth() — no custom session parsing. Phase 1 L
User rolesPermission model Roles (if any) are stored in the SQLite database and checked manually per route. No standardized role hierarchy. Use Clerk's organization roles: admin, producer, viewer. Roles are attached to the Clerk organization membership — no custom role table needed. Enforce role checks via Clerk's has() helper in middleware and API routes. Phase 1 M
Invite systemTeam member onboarding No invite system exists. New Workprint employees are added via Google Admin and can immediately sign in. There is no invite link, no invite email, and no pending invite state. Build invite flow: admin generates invite link → Resend email with link → recipient clicks → Clerk signs them up → they're added to the studio org automatically. Clerk's organization invitations API handles most of this. Store pending invites in the DB for UI display. Phase 1 L
Password resetAccount recovery Does not exist. Google handles passwords. There is no "forgot password" page, no reset email flow, no email verification on signup. Handled automatically by Clerk once the auth migration is complete. Clerk provides hosted flows for these at configurable URLs. Wire them to Smack's custom UI components to avoid breaking the design. Phase 1 S
03
Billing
Critical
Hard blocker. There is no subscription management, no payment collection, no plan limits, and no concept of a paid account. The app currently runs as internal tooling with no revenue infrastructure whatsoever.
Item Current State What's Needed Phase Complexity
Payment processingStripe integration None. Workprint is the only user of the app. No Stripe account, no payment collection, no subscription concept exists anywhere in the codebase. Integrate Stripe Billing. On studio creation: create a Stripe customer. On plan selection: create a Stripe subscription. Store stripe_customer_id and stripe_subscription_id on the studio record. Use Stripe Checkout for the initial payment flow — don't build a custom card form. Phase 1 XL
WebhooksSubscription lifecycle No webhook endpoint exists. Stripe subscription states (active, past_due, canceled, trialing) are not tracked. Build a /api/webhooks/stripe endpoint. Handle: checkout.session.completed, invoice.payment_succeeded, invoice.payment_failed, customer.subscription.deleted. Update studio plan_status on each event. Use Stripe's webhook signature verification — never trust unverified webhook payloads. Phase 1 L
Plan limitsFeature gating No plan limits exist. Users can create unlimited projects, invoices, and bookings. No concept of a free tier, paid tier, or usage ceiling. Define plan tiers (e.g., Starter: 3 active projects, 2 users; Pro: unlimited). Enforce limits in API routes before resource creation. Return a 402 with an upgrade prompt when limits are hit. Limits are stored per-plan in a config file — not hard-coded in every route. Phase 1 M
Grace periodFailed payment handling No grace period logic. No delinquency state, no access restriction, no retry flow. When invoice.payment_failed is received: set studio status to past_due. Show an in-app banner. Give a 7-day grace period before restricting to read-only. Send reminder emails via Resend at day 1, 3, and 7. On day 7, set status to suspended — read-only, no new resource creation. Phase 1 M
CancellationSubscription management No cancellation flow, no subscription management page, no way to downgrade or cancel from within the app. Build a Studio Settings → Billing page: shows current plan, next billing date, payment method, and a "Cancel subscription" option. Cancellation via Stripe API sets to cancel at period end (not immediately). Show data export prompt at cancellation. Phase 2 M

Studio Setup & Onboarding
High
Item Current State What's Needed Phase Complexity
Studio creation flowFirst-run experience No studio creation flow exists. The app boots directly into Workprint's data — there is no concept of "setting up a new studio." Full onboarding wizard (see onboarding flow doc): studio name → timezone → logo → invite team → first project. The first time an admin signs up, they must complete this flow before reaching the dashboard. Non-admins joining an existing studio skip the studio setup steps. Phase 1 L
Studio settings panelAdmin controls No studio settings page. There is no place to change the studio name, upload a logo, set a timezone, or manage billing. Workprint's settings are hardcoded or managed outside the app. Build a Settings panel with tabs: General (name, logo, timezone), Team (invite, manage roles, remove members), Billing (plan, payment method, invoices). Only users with the admin role can access Settings. Route: /settings. Phase 1 L
Team member managementRoster UI No team management UI. Workprint staff are managed via Google Admin. There is no in-app list of team members, no way to change roles, and no way to remove someone. Settings → Team page: shows all studio members with their role, last active date, and a "Remove" action. Shows pending invites with a "Resend" or "Cancel" option. Admins can change any member's role. Role changes update both the Clerk organization membership and the local DB. Phase 1 M
Studio slug / URLRouting identity All routes are at the root level — projects.workprint.la/projects. There is no studio-specific URL segment. Each studio gets a unique slug set at creation (e.g., rooftop → app.smack.studio/rooftop/projects). Slug is used in all authenticated routes and in external links (contractor portal, client portal). Studio can rename slug in settings with a redirect from the old one. Phase 2 M
05
Workprint-Specific Integrations
High
Item Current State What's Needed Phase Complexity
LucidLink integrationFile storage Hardcoded to Workprint's specific LucidLink filespace. Creates project folders automatically at Workprint's defined path structure (/Workprint/Clients/[client]/[project]). LucidLink API key is stored as a production environment variable. Remove all LucidLink code from the core app. Extract into a studio-specific integration module that can be enabled per-studio via Settings → Integrations. When enabled for a studio, they configure their own LucidLink API key and base path. All other studios are unaffected. Phase 1 L
Google Admin integrationContractor provisioning When a contractor is onboarded, the app automatically creates a @workprint.la Google Workspace account for them via the Google Admin SDK. This is entirely Workprint-specific behavior. Remove the Google Admin auto-provisioning entirely. Contractors access Smack via their own email and a contractor portal invite. No external account provisioning should be done by the app on behalf of any studio. Remove all googleapis admin SDK references from the codebase and CI environment. Phase 1 M
Hardcoded org referencesCode-level coupling References to "Workprint" exist in: API response copy, email templates, PDF templates, error messages, and comments throughout the codebase. Several environment variables reference workprint-specific infrastructure. Full codebase grep for "workprint" (case-insensitive). Every hardcoded reference must be replaced with a dynamic studio name pulled from the authenticated session's studio record. Create a helper getStudio() function used consistently across server components and API routes. Phase 1 M
06
Email & Notifications
High
Item Current State What's Needed Phase Complexity
Sender domainEmail identity All transactional emails send from noreply@workprint.la via Resend. The Resend sender domain is verified as workprint.la. No other domain is configured. Verify smack.studio as a Resend sender domain. Change the default system sender to noreply@smack.studio. Allow studios to configure a custom "reply-to" address (e.g., their own email) in Settings → General. Contractor and client-facing emails should reply-to the studio's address, not Smack's. Phase 1 M
Email templatesCopy & branding All email templates reference "Workprint" by name. They include the Workprint logo, Workprint color scheme, and Workprint-specific instructions (e.g., "check your LucidLink for project files"). Rebuild all email templates to be studio-neutral. Header should show the sending studio's name and logo (pulled from their studio record). Footer: "Sent via Smack" — not "Workprint." Remove all Workprint-specific copy from template bodies. Use React Email or equivalent for maintainable templates. Phase 1 L
Notification typesCoverage Limited to a few manual triggers: booking confirmation, invoice sent. No notifications for: project status changes, team invites, payment received, booking reminders. Extend notification coverage for launch: team invite, invoice viewed, invoice paid, booking confirmation, project created, password reset. Each notification type should be individually toggle-able per user in their profile settings. Don't send emails users didn't ask for. Phase 2 L
07
Contractor Portal
High
Item Current State What's Needed Phase Complexity
Portal domain / URLAccess point The contractor portal lives at portal.workprint.la. This is a separate Next.js app (or subdomain route) hard-coded to the Workprint domain and identity. Migrate to portal.smack.studio. Portal must be studio-aware — when a contractor clicks their invite link, the portal context is scoped to the inviting studio (logo, name, color if customized). The URL can remain a Smack subdomain as long as the studio is clearly identified on-screen. Phase 1 M
Onboarding stepsContractor setup flow Contractor onboarding includes a LucidLink invite step ("Accept your LucidLink invite from Workprint") and references Workprint's specific Google Drive folder structure. These steps are hard-coded. Remove LucidLink step from default contractor onboarding. The step should only appear if the inviting studio has LucidLink enabled in their integrations. Replace "Workprint" references with the dynamic studio name. Remove all Google Drive references — these are Workprint-specific workflows. Phase 1 M
Auth for contractorsPortal access model Contractors receive a @workprint.la Google account and sign in via Google OAuth. No other auth method exists for contractor portal access. Contractors must be able to access the portal with their own personal or work email. Auth via: email/password or Google OAuth on their own account. Clerk handles this. A contractor's portal access is scoped to the specific bookings/invoices shared with them by any studio — not a full login to the studio app. Phase 1 L
Contractor data scopeWhat contractors see Currently unknown whether contractors can see data from other contractors or other projects. Given the single-tenant architecture, this is likely an open issue. A contractor in the portal can only see their own bookings, their own invoices, and the project details explicitly shared with them. They cannot browse the studio's project list, crew roster, or financial data. Enforce this via Clerk role contractor and explicit resource-level permission checks. Phase 1 M
Medium Priority — Required Before Wide Availability

Client-Facing Content
Medium
Item Current State What's Needed Phase Complexity
Invoice PDFsClient-facing documents Invoice PDFs are generated server-side and include the Workprint name, Workprint logo, and Workprint contact information in the header. A client receiving an invoice from another studio would see Workprint branding, not their vendor's. Invoice PDF template must pull studio name, logo, address, and contact details from the studio record. Client only sees their vendor's branding. No mention of Smack on client-facing PDFs — the platform should be invisible to end clients. PDF footer can include "Powered by Smack" in tiny gray text as an optional opt-in. Phase 1 M
Review app / approval linksExternal sharing Review links reference workprint.la in their URLs and page copy. Clients clicking a review link would see Workprint-branded pages regardless of which studio sent the link. Review/approval pages must show the sending studio's name and logo. URL can remain on Smack's domain (review.smack.studio/[token]) but the page identity should reflect the studio. No hardcoded Workprint references on any publicly accessible URL. Phase 2 M
Client portalExternal access If a client portal exists, it is at a Workprint domain and references Workprint throughout. Its multi-studio readiness is unknown. Audit client portal for Workprint references. Apply same studio-scoping pattern as contractor portal — client can only see their own projects and shared materials. Client portal URL should be studio-slug-based or token-based, not Workprint-branded. Phase 2 L
09
Data Export & Privacy
Medium
Item Current State What's Needed Phase Complexity
Data exportStudio data ownership No data export functionality exists. Studios using Smack must trust that their data is accessible via the UI indefinitely. If the studio cancels or if Smack shuts down, there is no way to extract data. Settings → Data Export: generate a ZIP containing CSVs for all projects, bookings, invoices, and crew members. Export should be available at any time, not just on cancellation. Add an export trigger automatically at cancellation with a 30-day data retention window after account closure. Phase 2 M
Account deletionRight to erasure No account deletion flow. No studio deletion. No way for a user to request erasure of their personal data. This is a GDPR requirement for any studio with EU clients or crew. Build: user-level account deletion (removes personal data, anonymizes authored records), studio-level deletion (requires admin, triggers data export prompt, 30-day deferred deletion). Implement a deleted_at soft-delete pattern. Hard delete after 30 days via a scheduled job. Phase 2 L
Privacy policy & termsLegal pages No privacy policy page, no terms of service. These pages don't exist at any URL in the current app. Write and publish Privacy Policy and Terms of Service at smack.studio/privacy and smack.studio/terms. Link from the signup flow, footer, and email templates. Use a standard SaaS privacy policy template reviewed by a lawyer before any paying customers sign up. Phase 2 S
Low Priority — Phase 3 / Post-Launch Polish
10
Help & Support
Low
Item Current State What's Needed Phase Complexity
Help documentationSelf-serve answers No help documentation exists. Users with questions about how the app works have no reference. Internal Workprint users could ask each other — external users cannot. At minimum: a simple static HTML help page at smack.studio/help covering the 10 most common questions (how to create a project, how to send an invoice, how to invite a contractor, etc.). Can evolve to a Notion-embedded docs site or a proper help center (Intercom, Crisp) over time. Phase 3 S
In-app supportContact channel No support widget, no contact email displayed in-app, no error reporting. If something breaks, users have no path to get help other than finding the company somehow. Minimum viable: a "Get help" link in the app sidebar that opens a mailto to support@smack.studio. Better: Crisp or Intercom chat widget installed on all authenticated pages. Don't gate support behind a separate portal — make it one click from anywhere in the app. Phase 3 S
Error pages404 / 500 handling Next.js default error pages are shown. The 404 page shows generic Next.js styling with no navigation, no branding, and no helpful context. 500 errors show a blank screen. Custom not-found.tsx and error.tsx pages in the Next.js app directory. 404: show the Smack logo, a friendly message, a "Go home" button, and a "Contact support" link. 500: same, plus an automatic error report sent to Sentry (or equivalent). These pages take 2 hours to build — there's no reason to ship without them. Phase 3 S
