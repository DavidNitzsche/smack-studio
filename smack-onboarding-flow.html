Key Design Decision
The studio logo/initials and studio name appear above the fold, immediately. The invite is from a person they know, to a studio they work with — make that social proof the hero. Don't lead with Smack branding here; lead with the studio identity.
Why Two CTAs?
"Set up later" accepts the invite and creates the account shell without forcing them through the full flow. This preserves the invite acceptance even if they close the tab. Don't make them lose their invite by not setting up immediately.
What Can Go Wrong
Expired invite links. Make the expiry window generous (7 days minimum) and show a clear re-send prompt if expired. Don't show a generic 404.
Role Shown On This Screen
Role is shown in a pill so they know what access they're getting before accepting. Reduces "wait, what am I signing up for?" hesitation.

Key Design Decision
Google OAuth is the primary CTA, positioned above the form. Most production studio folks use Google Workspace. Don't make them type a password if they don't have to — reduces time and friction to near zero.
Pre-filled Email
The email from the invite is pre-filled and read-only (grayed out). This confirms they're setting up the right account and prevents them from accidentally using a different email address that won't match the invite.
No Marketing Here
This screen is pure function. No feature bullets, no pricing callouts, no testimonials. They already said yes. The only job of this screen is to get them through auth as fast as possible.
Clerk Note
Clerk handles email verification, password strength, and OAuth flows. Custom UI via Clerk's appearance API to match Smack's design system. Don't use Clerk's hosted pages for this step — it breaks the flow continuity.

Key Design Decision
"Skip for now" must be prominent — not a tiny gray link buried at the bottom. A logo upload is nice to have, not required. If you gate the flow behind optional fields, you lose people.
Progress Bar
Show a progress bar during the admin onboarding steps (Steps 3–5). This sets expectations — they know there are 4 steps and they're on step 1. Don't show it during invite-only flows.
Why Only 3 Fields?
Studio name and timezone are the only two fields with real product impact on day one. Name appears in all UI. Timezone affects project scheduling and booking displays. Logo is cosmetic — defer it.
Non-Admin Routing
Non-admin team members skip this step entirely. Their role is already set by the invite. They jump from Step 2 directly to the Dashboard (Step 7). No unnecessary setup for people who don't own the studio.

Key Design Decision
"Skip for now" has an arrow. It's not a failure state — it's an expected action. The arrow signals forward motion. Don't make skipping feel like giving up.
Tag-Style Email Input
Multi-email entry uses a tag input (type email, press Enter to add). This is faster than clicking "Add another row" repeatedly. Paste a comma-separated list and it auto-splits.
Role Assignment
All invites in a batch share one role. If you need different roles, use "Add more" to create additional rows. Keep the common case (one role, multiple people) as simple as possible.
Invites Fire in Background
After clicking "Send invites," the emails queue and the user immediately moves to the next step. Don't block on email delivery. Show a "Invites sent!" toast at the top of the next screen.

Create First Project

This is the Time-to-Value moment. Every extra field here is a conversion killer. 4-5 fields max.
60 – 90 seconds Drop-off: Medium

First project Step 3 of 4

Key Design Decision
Exactly 5 fields. Non-negotiable. Project name, client name, start date, end date, phase. That is the minimum viable data to make a project useful. Budget, team, deliverables, briefs — all of that comes after they're in.
No Native Date Pickers
Date fields use a custom popover calendar (not the browser's native date input). Native date pickers look different on every OS, are impossible to style, and create layout jitter. Use a lightweight custom picker.
No Skip Option Here
This is the only step with no skip. You can't activate on Smack without at least one project. But the fields are so few that no one should need to skip. If they're bailing here, the fields are wrong — audit them, don't add a skip.
Activation Gate
Project created = activated user. This is the metric that matters for onboarding success. Everything in Steps 1–4 exists to get someone to this moment cleanly. Track completion rate of this step above all others.

Key Design Decision
The project name from Step 5 is used verbatim in the heading: "Who's on Spring Brand Campaign?" This creates continuity and context. They're no longer doing setup — they're doing their actual job.
Pre-added: Themselves
The admin who created the project is automatically added as the first crew member. Their row appears at the top of the list with a default role and the full project date range. They just need to confirm or edit — not start from scratch.
Search + Invite Combined
The crew search field accepts either an existing studio member (autocomplete from the team roster) or a new email address to send an invite. One field, two behaviors — no modal required.
Skip Language
"Skip — go to my dashboard" is explicit about the destination. After all this setup, they deserve to know the journey is about to end. Don't use just "Skip" — they need the reward framing.

Key Design Decision
No welcome modal. Modals at this point in the flow feel like the product is insecure — like it needs to explain itself. The project they just created is already visible on the dashboard. That is the product working. Let it speak for itself.
The Checklist Widget
A small, non-blocking widget in the lower-right corner (not a sticky banner, not a modal). "Create first project" is already checked. Two nudges remain: add an invoice, invite a client. This drives the next activation milestones passively.
Project Already Checked
The checklist shows the project creation already done and checked. This creates an immediate sense of progress and momentum. Start with a win. Don't show a blank unchecked list on first login.
No Forced Product Tour
No tooltip chains, no "click here next" overlays. Those tours have terrible completion rates and create anxiety. The UI should be discoverable on its own. If it's not, the UI needs to be redesigned — not covered in tooltips.

Top 3 Flow Killers

1
Too many fields in Step 5 Every field beyond 5 in the project creation form reduces completion. Budget, status, description, department — none of these belong here. They belong in the project edit view after the user is already in the product.
2
Auth friction in Step 2 If Google OAuth doesn't work or email verification emails land in spam, the user is stuck. Test deliverability from Clerk's domains on GSuite, Outlook, and iCloud accounts. Every failed auth delivery is a lost activation.
3
A welcome modal at Step 7 Showing a modal, tour, or overlay when the user finally reaches their dashboard tells them the product isn't self-explanatory. It creates anxiety instead of confidence. Kill the modal. Trust the UI.
