01 / The Vision

Built by a studio,
for studios.

Smack exists because the software that production studios actually need doesn't exist. You built the right tool for yourself — now the question is how to make it work for everyone else.

The premise

Production studios don't need another generic project management tool. They need something that speaks their language — projects and phases, crew booking, invoicing in and out, contractor portals, and cash flow forecasting — all in one place, without a 6-month implementation or a $50k annual contract.

The name

A smack is the collective noun for a group of jellyfish — and like a production studio, a smack moves together, reacts together, and has no central brain telling each one what to do. The name is punchy, memorable, and completely ownable in a category full of generic names like "Flow" and "Studio" and "Manage."

06 / What Needs to Change

The delta between
internal tool and product.

The app is production-grade and battle-tested. But it was built for one studio. Going multi-tenant requires specific, concrete changes — none of which are rewrites. Here's every one of them.

Multi-Tenancy Architecture
High Effort Critical
Currently the app runs against a single SQLite database with no concept of "which studio does this data belong to." Every project, booking, invoice, and contractor is implicitly Workprint's. For Smack, each studio's data must be completely isolated.
Recommendation: One SQLite database file per studio, stored at /data/{studio_id}.db
Why: Keeps existing stack, perfect data isolation, simple backup (copy one file), easy to delete a studio, scales well to hundreds of tenants on Railway volumes.
Alternative: Single PostgreSQL DB with studio_id on every table — more complex, harder to guarantee isolation, but better for very large scale (10k+ studios).
Verdict: Start with per-studio SQLite. Migrate to Postgres when it's a problem worth having.
Authentication — Remove Google Workspace Lock-In
High Effort Critical
Currently auth requires a @workprint.la Google account — completely non-starter for other studios. Replace with a real auth layer supporting email/password, Google OAuth, and magic links.
Recommended: Clerk.com — handles auth, user management, org/multi-tenant by default. $0 up to 10k MAU, then $25/mo. Integrates cleanly with Next.js via middleware.
Alternative: Better Auth (open source, self-hosted) — free but requires maintenance.
Role system needed: Owner / Admin / Member / Viewer per studio.
Contractor portal auth stays separate — keep the existing token-based invite flow, just adapt it to per-studio context.
Studio Signup & Onboarding Flow
Medium Effort Critical
Right now there is no signup — you're handed credentials. For Smack, a new studio owner needs to go from landing page to their first project in under 5 minutes. This requires a new onboarding flow.
Flow: Sign up → Name your studio → Invite team (optional, skippable) → Create first project (guided) → You're in.
Studio setup: name, timezone, currency, logo upload (for PDFs and portal)
First project template: ask project name and phase, create it, land directly on the project page.
Goal: time-to-value under 5 minutes from signup.
Billing & Subscription Management
Medium Effort Critical
Stripe Billing integration. Subscription per studio, not per user. Handle trial periods, upgrades, downgrades, and cancellation. Studios pay one flat rate regardless of seat count within tier.
Stripe Products: Starter ($79/mo), Studio ($149/mo), Enterprise ($299/mo)
14-day free trial — no credit card required to start.
Webhook handling: subscription.created, subscription.deleted, invoice.payment_failed
Billing portal: link to Stripe's hosted portal for self-serve plan changes.
Cost to Smack: 2.9% + $0.30 per transaction (standard Stripe rate)
Branding & White-Labeling
Low Effort Critical
Remove all Workprint references. Replace with Smack branding. The contractor portal in particular should feel like it belongs to the studio using Smack, not to Smack itself — studios want their contractors to feel like they're working with them, not a third-party tool.
Remove: "Workprint" in all strings, emails, PDFs, page titles
Add: Smack logo, color tokens updated to Smack brand
Studio white-labeling (V2): portal header shows studio name + logo, email "from" name is the studio's
PDF invoices: show studio's logo/name (already works this way, just needs studio data instead of hardcoded WP values)
Subdomain routing: app.smack.studio (admin), portal.smack.studio (contractors)
Workprint-Specific Integrations
Low Effort Defer
Google Workspace account auto-provisioning and LucidLink folder creation are deeply specific to Workprint's infrastructure. Remove both for V1. Consider them as premium add-ons in a future integration marketplace.
V1: Remove both integrations completely. Contractors log in via Smack auth, not Google accounts. No file system provisioning.
V2 (optional add-ons): "Google Workspace" integration (for studios that want auto-provisioned accounts), "LucidLink" integration (for studios already using it), generic cloud storage hooks (Dropbox, Google Drive, Frame.io)
Framing: "Smack connects to your existing tools" — not "Smack does the tools for you"
Legal & Compliance
Low Effort Required
Can't go public without these. None of them require code — they're documents and configurations.
Privacy Policy — use Termly or Iubenda to generate ($15-25/mo, auto-updates for law changes)
Terms of Service — standard SaaS ToS template, have a lawyer review (~$300-500 one-time)
GDPR/CCPA — cookie banner (Cookiebot or CookieYes, free tier works), data deletion on request
Email compliance — Resend already handles SPF/DKIM, ensure unsubscribe links in marketing emails
Data processing agreement (DPA) — for EU studios, required under GDPR
Daily Automated Backups
Low Effort Non-Negotiable
Must be in place before the first real studio goes live. Since each studio is its own SQLite file, a backup is literally just copying a file — simple, fast, and cheap. If something breaks or data gets corrupted, you restore from yesterday in under a minute without touching any other studio's data.
How it works: Daily Railway cron job copies every /data/{studio_id}.db → uploads to Cloudflare R2 (S3-compatible) → keeps 30 days of rolling history.
Recovery: one command to restore a single studio file. No complex snapshot process, no collateral damage to other studios.
Cost: Cloudflare R2 free up to 10GB. At 10–50 studios the total DB size will be a few MB. Effectively free for a long time.
Monitoring: cron job sends a ping to a health-check URL (e.g. healthchecks.io — free) on success. If it goes silent, you get an email alert.
Email Infrastructure
Low Effort Straightforward
Currently all transactional emails come from @workprint.la via Resend. For Smack, system emails send from @smack.studio (or @mail.smack.studio). Studio-branded emails (invoices, booking confirmations) should appear to come from the studio.
Change: Update Resend sender domain to smack.studio
Invoice emails: "From: Acme Films via Smack <invoices@mail.smack.studio>" — studio name in the from display
Booking emails: same pattern — appears to come from the studio, handled by Smack
Cost: Resend free tier (3k emails/mo), Pro at $20/mo (50k emails). Plenty for early stage.

09 / Go-to-Market

Ship it. Put it out there.
See what happens.

This is a side project, not a startup sprint. The goal is to get it live, let it breathe, and let organic interest do the work. Low overhead, no pressure — but a few smart moves that cost almost nothing.

The approach

Build it well. Launch it quietly. Let the product speak. Production is a small world — if one studio owner finds it useful and tells another, that's how this grows. No ad spend, no cold outreach required. Just a real tool solving a real problem, available at smack.studio.

Instagram

Short-form content that shows the product in action. Screen recordings of real workflows — booking crew, generating an invoice, watching cash flow update. No need to post constantly — a few times a month of genuinely useful content beats daily filler. Tag production-adjacent accounts. Let it build slowly.

LinkedIn

Where studio owners actually are. Post from your personal account as the founder — not just from the Smack page. "I built this tool for my own studio and decided to make it available." That story is more compelling than any ad. Authentic, zero cost, and production people respond to it.

The marketing site does the heavy lifting

smack.studio is the always-on sales rep. Invite-only means anyone who finds it — from search, from a share, from a post — fills out a simple request form. You control who gets in, at what pace, and you get a direct line to every early user before they even log in. That's more valuable than open signups.

If it starts getting traction

That's the signal to open it up. Flip invite-only to open access, run a Product Hunt launch, lean into content. But that comes later and only if the data says so. Right now the only job is: build it, ship it, and let the right people find it.

The only metric that matters early

Invite request → accepted → paid conversion. If studios you let in end up paying, the product works and the market exists. If they don't, the answer is in the product — not the marketing. Keep it simple, watch that number, and let it tell you what to do next.

10 / Operations

Decisions made before
the first studio signs up.

These aren't open questions — they have industry-standard answers. Decide them now so they're baked into the product from day one, not retrofitted after a studio asks.

The Invite Request Form — What to Capture
Pre-launch
The invite form is your first conversation with a potential customer. Ask enough to qualify them and personalize their onboarding — but not so much that they bail before submitting.
Capture: Studio name · Your role (EP, Producer, Owner, etc.) · Estimated active projects per month (1–5 / 6–15 / 15+) · What they currently use (spreadsheets / Monday / other / nothing) · One sentence: what problem are you trying to solve?

Why this works: project volume tells you which tier they belong in. "What they currently use" is your best sales research. The open-ended question surfaces language you can use in marketing copy verbatim.

Standard: Keep it to 5 fields max. Under 60 seconds to complete. No phone number required.
Founding Studio Pricing — Lock It In for Life
Revenue lever
Studios who join in the first 6 months get their rate locked for as long as they stay subscribed. If you raise prices later, they never see it. This is standard SaaS practice and it works — it converts fence-sitters and creates extremely loyal early customers.
Industry standard: Basecamp, Linear, and dozens of others have done this. "Founding member" pricing as a permanent lock-in is a proven early-adopter conversion tool.

Suggested: First 10 studios pay $49 / $99 / $199 (vs. standard $79 / $149 / $299) — locked forever as long as they stay active.

Communicate it clearly: "You're joining as a founding studio. Your rate is locked for life." That sentence alone is worth more than any feature list.
Cancellation — Data Export & Retention
Industry standard
Studios will ask this before signing up, especially if they're migrating real financial data. The answer needs to be clear and reassuring.
Standard policy: On cancellation, studio retains full access for the remainder of their billing period. After that, the account is locked (read-only) for 30 days — data is not deleted.

Data export: CSV export of all projects, invoices, and bookings available any time, including after cancellation during the 30-day window. PDF invoices are always downloadable.

Permanent deletion: After 30 days of locked status, all data is permanently deleted. Studio receives an email warning 7 days before deletion.

Why this matters: Studios with 2 years of invoice history need to know they can leave without losing records. This removes the biggest psychological barrier to signing up.
Payment Failure — Grace Period
Industry standard
Cards expire, payments fail. How you handle this determines whether you lose the customer or just the payment.
Standard Stripe Smart Retries: Stripe automatically retries failed payments 3–4 times over 7 days using ML to pick optimal retry times (90%+ recovery rate).

Grace period: Full access maintained for 7 days after first failure. Day 1: email notifying of failure. Day 5: reminder with direct link to update card. Day 7: access suspended (not deleted). Day 14: final notice before cancellation.

Reinstatement: Updating payment reinstates access instantly — no re-onboarding required.

This is table-stakes SaaS behaviour. Stripe handles most of it natively — costs nothing to implement correctly.
Security — The One-Liner Answer
Studios will ask
Studios won't ask for a SOC 2 report — but they will ask where their data lives and who can see it. Have a clear, confident answer ready.
The answer: "Your data lives in an isolated database on Railway's infrastructure, hosted on AWS US-East. It is encrypted at rest and in transit. Only you and the team members you invite can access your studio's data. Smack staff cannot view your project or financial data. Daily backups are retained for 30 days."

At rest encryption: Railway volumes encrypt at rest by default (AES-256). Check this is confirmed in Railway docs — it is standard on AWS-backed infrastructure.

In transit: All traffic over HTTPS/TLS. Handled by Railway + Cloudflare.

Isolation: Per-studio SQLite means a bug in one studio's account cannot expose another's data. This is actually a stronger isolation model than shared-database SaaS and worth saying so.
The Accept Email — First Real Brand Touchpoint
High impact
When you approve an invite, the email that lands in that studio owner's inbox is the first time they experience Smack as a real product. It should feel like hearing from a person, not a system.
Tone: Warm, direct, brief. Not corporate. Written like it's from you personally — because it is.

Structure: Subject: "You're in." · Opening: one sentence welcoming them by name. · Their founding rate (if applicable), clearly stated. · One link to set their password and get started. · One sentence of what to do first ("Create your first project — it takes 90 seconds"). · Sign off with your name.

Do not: list features, explain pricing tiers, include legal disclaimers in the body, or use words like "onboarding journey".

This email sets the tone for the entire relationship. Studios who feel like they've been personally welcomed are dramatically less likely to churn in the first 30 days.

Why this matters

You are the ICP. If something annoys you as the operator, it will annoy every studio owner. If the invoice PDF looks wrong, you'll see it on a real invoice. If the cash flow forecast is confusing, you'll feel it while running an actual project. No amount of QA testing replicates the pressure of real work on a real deadline. Run your real business on it and the product will be better for every studio that follows.

The migration plan

Step 1 — Backups first. Do not move anything until the automated daily backup is running and verified. Pull a manual backup of the current Workprint DB and confirm you can restore it.

Step 2 — Parallel run (1 week minimum). Run both the old Workprint app and Smack simultaneously. Enter new projects into Smack. Keep old data accessible but don't add to it. Watch for anything that doesn't feel right.

Step 3 — Import historical data. Migrate existing projects, invoices, and contractors via a one-time import script. Verify everything looks correct before cutting over.

Step 4 — Cut over. Point the old Workprint app URL to Smack. Archive the old codebase but don't delete it for 30 days.

What you're testing

Every part of the product that a real studio owner will touch: signing up (do it as if you don't know the system), creating a project, booking a contractor (have a real contractor go through the portal), sending an invoice to a real client, checking cash flow at the end of the month. Every friction point you find is one less reason for a future studio to churn.

One important rule
Non-negotiable
Pay the $149/month. Don't exempt Workprint from billing. The moment you stop paying, you stop feeling what your customers feel. Price sensitivity, billing emails, renewal notices — all of it matters. Workprint paying Smack is also your first real MRR data point. The day you invoice yourself is the day Smack becomes a real business.
